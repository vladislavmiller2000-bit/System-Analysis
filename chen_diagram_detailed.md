# ER-диаграмма в нотации Чена: Полная версия со всеми элементами

## 1. СУЩНОСТИ (Entities)

### 1.1 Независимая сущность (Independent Entity)
```
┌──────────────────┐
│     ЗАКАЗ        │
└──────────────────┘
```
- **Обозначение**: Прямоугольник с одной рамкой
- **Описание**: Существует в системе всегда, независимо от связей с другими сущностями
- **Пример**: ЗАКАЗ - это основная сущность интернет-магазина, которая всегда существует в системе

### 1.2 Зависимая сущность (Dependent Entity)
```
╔══════════════════════╗
║ ТОВАР В ЗАКАЗЕ      ║
║ СВЕДЕНИЯ О ДОСТАВКЕ ║
╚══════════════════════╝
```
- **Обозначение**: Прямоугольник с двойной рамкой
- **Описание**: Зависит от другой сущности и не может существовать без неё
- **Примеры**:
  - ТОВАР В ЗАКАЗЕ - зависит от ЗАКАЗА
  - СВЕДЕНИЯ О ДОСТАВКЕ - зависит от ЗАКАЗА

### 1.3 Ассоциативная сущность (Associative Entity)
```
╔──────────────────────╗
║ АССОЦИАТИВНАЯ        │
║ СУЩНОСТЬ             │
╚──────────────────────╝
```
- **Обозначение**: Ромб или прямоугольник с двойной рамкой (в зависимости от типа)
- **Описание**: Создаётся для разрешения отношения M:N между двумя сущностями
- **Пример**: Если бы была связь M:N между ЗАКАЗОМ и ТОВАРОМ, можно было бы создать ассоциативную сущность ПОЗИЦИЯ_ЗАКАЗА

## 2. АТРИБУТЫ (Attributes)

Все атрибуты имеют форму **овала** и соединяются прямыми линиями с сущностью.

### 2.1 Первичный ключ (Primary Key)
```
      ┌─────────────┐
      │ № заказа    │ ← Подчеркнутый текст
      │    (PK)     │
      └──────┬──────┘
             │
        ─────┴─────
             │
         ┌───────┐
         │ ЗАКАЗ │
         └───────┘
```
- **Обозначение**: Овал с **подчеркнутым названием**
- **Описание**: Однозначно идентифицирует один экземпляр сущности
- **Свойства**:
  - Должен быть уникальным
  - Не может быть NULL
  - В сущности может быть только один простой первичный ключ
- **Пример**: № заказа - каждый заказ имеет уникальный номер

### 2.2 Множественный атрибут (Multi-valued Attribute)
```
      ┌──────────────┐
      │ Товары в     │ ← Двойной овал
      │ заказе       │
      │(Множественный)
      └──┬──────────┬┘
         │          │
     ────┴──────────┴────
             │
         ┌───────┐
         │ ЗАКАЗ │
         └───────┘
```
- **Обозначение**: **Двойной овал** (овал в овале)
- **Описание**: Атрибут, который может иметь несколько значений для одного экземпляра сущности
- **Пример**: Товары в заказе - один заказ может содержать несколько товаров
- **Связь с сущностью**: Утолщённая линия (линия с кратностью 1:N)

### 2.3 Производный атрибут (Derived Attribute)
```
      ┌─────────────┐
      │ Сумма       │ ← Пунктирный овал
      │ заказа      │
      │(Производный)│
      └─────┬───────┘
            │ (пунктирная линия)
        ┌───────┐
        │ ЗАКАЗ │
        └───────┘
```
- **Обозначение**: Овал с **пунктирной линией контура**
- **Описание**: Атрибут, который вычисляется или выводится из других атрибутов
- **Свойства**:
  - Не хранится в базе данных
  - Вычисляется при необходимости
- **Примеры**:
  - Сумма заказа = Σ(Цена × Количество × (1 - Скидка))
  - Сумма скидок = Σ(Размер скидки)

### 2.4 Составной атрибут (Composite Attribute)
```
         ┌──────────────────┐
         │ Адрес доставки   │ ← Простой овал
         └────┬────────┬──┬─┘
              │        │  │
         ┌────┴──┐ ┌──┴─┐ └──────────┐
         │Страна │ │Город│ Улица ...  │
         └───────┘ └────┘ └──────────┘
              │
          ┌───────┐
          │ ЗАКАЗ │
          └───────┘
```
- **Обозначение**: Овал (как простой атрибут), но с дополнительными овалами-компонентами
- **Описание**: Атрибут, который состоит из нескольких податрибутов
- **Примеры**:
  - **Адрес доставки**: Страна, Город, Улица, Номер дома, Квартира, Почтовый индекс
  - **Сведения о доставке**: Способ доставки, Дата доставки
- **Связь компонентов**: Соединяются **одиночными линиями** с родительским атрибутом

### 2.5 Внешний ключ (Foreign Key)
```
      ┌──────────────┐
      │ ID           │ ← Обозначение (FK)
      │ Пользователя │
      │   (FK)       │
      └──────┬───────┘
             │
        ─────┴─────
             │
         ┌───────┐
         │ ЗАКАЗ │
         └───────┘
```
- **Обозначение**: Овал с пометкой (FK)
- **Описание**: Атрибут, который ссылается на первичный ключ другой сущности
- **Пример**: ID Пользователя ссылается на PK сущности ПОЛЬЗОВАТЕЛЬ

## 3. СВЯЗИ / ОТНОШЕНИЯ (Relationships)

### 3.1 Типы связей (Cardinality)

#### 3.1.1 Идентифицирующая связь (Identifying Relationship)
```
  ЗАКАЗ                содержит            ТОВАР В ЗАКАЗЕ
┌────────┐──────┐    ┌──────────┐    ┌──────────────┬─────────┐
│        │      │    │          │    │              │         │
│        │◄═════╡══════════════════════════════════╡          │
│        │      │    │          │    │              │         │
│        │      │    └──────────┘    │              │         │
└────────┘──────┘                    └──────────────┴─────────┘
         │                                       │
         └───────────────────────────────────────┘
              (Двойная линия = Идентифицирующая)
                         1        :        N
```
- **Обозначение**: **Двойная линия** от одной сущности к другой
- **Описание**: Каждый экземпляр одной сущности ДОЛЖЕН быть связан с экземпляром другой сущности
- **Свойства**:
  - Связь обязательна (mandatory)
  - Каждый экземпляр-ребёнок идентифицируется своим ID + ID родителя
- **Пример**: 
  - ТОВАР В ЗАКАЗЕ не может существовать без ЗАКАЗА
  - Полный ключ: (№ заказа, Артикул товара)

#### 3.1.2 Неидентифицирующая связь (Non-identifying Relationship)
```
  ПОЛЬЗОВАТЕЛЬ           оформляет           ЗАКАЗ
┌──────────────┐         ┌────────┐    ┌───────────┐
│              │    ┌────│        │────┤           │
│              ├─────┤    └────────┘    │           │
│              │    └────────────────────│           │
└──────────────┘                        └───────────┘
                    (Одиночная линия = Неидентифицирующая)
                              1    :    N
```
- **Обозначение**: **Одиночная линия** от одной сущности к другой
- **Описание**: Экземпляр одной сущности может существовать и БЕЗ связи с другой сущностью (не обязательна)
- **Свойства**:
  - Связь опциональна (optional)
  - Ребёнок имеет самостоятельный ID
- **Пример**: 
  - ПОЛЬЗОВАТЕЛЬ может существовать, даже если он не оформил ЗАКАЗ

### 3.2 Кардинальность связи (Cardinality Ratio)

#### 3.2.1 Один к одному (1:1)
```
СУЩНОСТЬ_A          имеет          СУЩНОСТЬ_B
┌──────────┐         ┌──────┐    ┌──────────┐
│          ├────────│      │────┤          │
│          │        └──────┘    │          │
└──────────┘        1     :     1 └──────────┘
```
- **Кардинальность**: Один экземпляр A связан с РОВНО одним экземпляром B
- **Пример**: ПАСПОРТ имеет ВЛАДЕЛЬЦА (один паспорт - один владелец, один владелец - один паспорт)
- **Запись связи**: A(1) ──── (1)B

#### 3.2.2 Один ко многим (1:N)
```
ЗАКАЗ               содержит         ТОВАР В ЗАКАЗЕ
┌────────┐         ┌──────────┐    ┌──────────────┐
│        │         │          │    │              │
│        ├────────│          │────┤              │
│        │        └──────────┘    │              │
│        │    (один заказ         │              │
│        │  содержит много        │              │
└────────┘     товаров)           │              │
   1              :              N└──────────────┘
```
- **Кардинальность**: Один экземпляр A связан с МНОГИМИ экземплярами B
- **Пример**: 
  - Один ЗАКАЗ может содержать много ТОВАРОВ В ЗАКАЗЕ
  - Один ПОЛЬЗОВАТЕЛЬ может оформить много ЗАКАЗОВ
- **Запись связи**: A(1) ───── (N)B
- **Связь часто идентифицирующая** (двойная линия)

#### 3.2.3 Многие ко многим (M:N)
```
ЗАКАЗ                  должен         ТОВАР
┌────────┐            ┌──────┐    ┌──────────┐
│        │   ┌───────│      │────┤          │
│        ├──┤│       └──────┘    │          │
│        │   └───────────────────┤          │
└────────┘           M    :    N └──────────┘
 
  (требует АССОЦИАТИВНОЙ СУЩНОСТИ)
  
      ЗАКАЗ              ПОЗИЦИЯ        ТОВАР
     ┌─────┐            ┌──────┐      ┌──────┐
     │     ├────┬───────│      ├──────┤      │
     │     │    │       └──────┘      │      │
     └─────┘    └────────┬────────────┤      │
      (1)        (N)     │    (N) (1) │      │
                   (Ассоциативная)   │      │
                                    └──────┘
```
- **Кардинальность**: МНОГО экземпляров A связано с МНОГО экземплярами B
- **Решение**: 
  - Если есть связь M:N, её разлагают на две связи 1:N через **ассоциативную сущность**
  - Ассоциативная сущность содержит внешние ключи обеих сущностей
- **Пример**:
  - ЗАКАЗ и ТОВАР имели бы связь M:N (один заказ содержит много товаров, один товар может быть в много заказах)
  - Решение: создаём ассоциативную сущность ПОЗИЦИЯ ЗАКАЗА (или ТОВАР_В_ЗАКАЗЕ)

## 4. ОБОЗНАЧЕНИЯ НА ЛИНИЯХ СВЯЗЕЙ

```
     ┌──────────────┐
     │ СУЩНОСТЬ А   │
     │              │
     └──────┬───────┘
            │
    ║ ║ или ─ или ╌ ╌ (в зависимости от типа)
    (двойная)  (одиночная) (с дополнит. смыслом)
            │
     1, N, M или N (кардинальность)
            │
     ┌──────┴───────┐
     │ СУЩНОСТЬ Б   │
     │              │
     └──────────────┘
```

### Типы линий:
1. **═══ (Двойная линия)** - Идентифицирующая связь (обязательная)
2. **─── (Одиночная линия)** - Неидентифицирующая связь (опциональная)
3. **╌ ╌ (Пунктирная линия)** - Производные/условные отношения

### Кардинальность:
- **1** - ровно один (на стороне "один")
- **N** - много (на стороне "много")
- **M** - много (альтернативное обозначение вместо N)
- **0..1** - ноль или один
- **1..* или 0..*** - более формальные записи

## 5. ПРАКТИЧЕСКИЙ ПРИМЕР: НАША ДИАГРАММА

### Структура:

```
                    ┌──────────────────┐
                    │   ПОЛЬЗОВАТЕЛЬ   │
                    │ (Независимая)    │
                    └────────┬─────────┘
                             │ (FK: ID Пользователя)
                             │ (M:1 неидентифицирующая)
                             │
                 ┌───────────┴────────────┐
                 │                        │
          ┌──────▼───────────┐    ┌──────┴──────────────┐
          │      ЗАКАЗ       │    │ ДОПОЛНИТЕЛЬНЫЕ      │
          │ (Независимая)    │    │ СУЩНОСТИ            │
          └──────┬───────────┘    │ (для расширения)    │
                 │                └─────────────────────┘
        ┌────────┼────────┐
        │ (1)    │        │ (1)
    ┌───▼───────┐ ▼      │
    │ содержит  │╔════════╡╗
    │           ║          ║
    │  имеет    ║          ║
    └───┬───────┘║        ║
        │        ║         ║
        │ (N)    ║  (1)   ║ (1)
    ┌───▼────────╨─┐  ╔════╬════╗
    │ ТОВАР В      │  ║    │    ║
    │ ЗАКАЗЕ       │  ║    │    ║
    │(Зависимая)   │  ║    │    ║
    └──────────────┘  ║    │    ║
                      ║    │    ║
              ┌───────╜────┘    ║
              │                 │
          ┌───▼──────────────────▼──┐
          │ СВЕДЕНИЯ О ДОСТАВКЕ     │
          │ (Зависимая)             │
          └────────────────────────┘
```

### Описание связей:

1. **ЗАКАЗ содержит ТОВАР В ЗАКАЗЕ**
   - Тип: **Идентифицирующая** (двойная линия)
   - Кардинальность: **1:N**
   - Описание: Один заказ содержит много товаров, каждый товар в заказе принадлежит ровно одному заказу
   - ТОВАР В ЗАКАЗЕ не может существовать без ЗАКАЗА

2. **ЗАКАЗ имеет СВЕДЕНИЯ О ДОСТАВКЕ**
   - Тип: **Идентифицирующая** (двойная линия)
   - Кардинальность: **1:1**
   - Описание: Один заказ имеет ровно одно сведение о доставке
   - СВЕДЕНИЯ О ДОСТАВКЕ не может существовать без ЗАКАЗА

3. **ПОЛЬЗОВАТЕЛЬ оформляет ЗАКАЗ** (при расширении модели)
   - Тип: **Неидентифицирующая** (одиночная линия)
   - Кардинальность: **1:N**
   - Описание: Один пользователь может оформить много заказов
   - ЗАКАЗ может существовать без явной ссылки на ПОЛЬЗОВАТЕЛЯ в этой версии

## 6. ЗАМЕЧАНИЯ И ДОПУЩЕНИЯ

### По поводу нотации Чена в нашей диаграмме:

1. **Множественный атрибут "Товары в заказе"**
   - Это может быть интерпретировано как связь 1:N к сущности ТОВАР В ЗАКАЗЕ
   - Двойной овал показывает, что это множественный атрибут

2. **Составные атрибуты**
   - Адрес доставки содержит компоненты
   - Сведения о доставке тоже содержат компоненты
   - Они показаны как овалы-компоненты под родительским атрибутом

3. **Производные атрибуты**
   - Сумма заказа и Сумма скидок вычисляются
   - Показаны пунктирными овалами

4. **Расширение модели**
   - Можно добавить сущность ПОЛЬЗОВАТЕЛЬ с неидентифицирующей связью к ЗАКАЗУ
   - Можно добавить сущность ТОВАР и связь M:N между ЗАКАЗОМ и ТОВАРОМ через ТОВАР_В_ЗАКАЗЕ
   - Можно добавить сущность АКЦИЯ и связь к ТОВАРУ_В_ЗАКАЗЕ

## 7. ИНСТРУКЦИИ ДЛЯ DIAGRAMS.NET

### Создание основной сущности:
1. Вставить **прямоугольник** (Rectangle)
2. Написать "ЗАКАЗ"
3. Выбрать цвет светло-жёлтый (#FFF9E6)
4. Размер: примерно 200-250px × 100px

### Создание зависимой сущности:
1. Вставить **прямоугольник** (Rectangle)
2. Написать название (ТОВАР В ЗАКАЗЕ, СВЕДЕНИЯ О ДОСТАВКЕ)
3. Установить **двойную рамку**: Format → Line → Width → 3px
4. Выбрать цвет светло-красный (#FFE8E8)

### Создание атрибутов:
1. Вставить **эллипс** (Ellipse)
2. Написать название
3. Соединить **одиночной линией** с сущностью
4. Для первичного ключа: подчеркнуть текст (Ctrl+U)
5. Для множественного: вставить два эллипса (внешний и внутренний, как вложенные)
6. Для производного: установить пунктирный контур (Format → Line → Pattern → Dashed)
7. Для составного: добавить компоненты и соединить одиночными линиями

### Создание связей:
1. Вставить **ромб** (Diamond) - символ связи
2. Написать название отношения (содержит, имеет)
3. Соединить с сущностями:
   - **Двойной линией** для идентифицирующей
   - **Одиночной линией** для неидентифицирующей
4. Добавить **кардинальность** (1, N) рядом с линиями

### Оформление:
- Использовать разные цвета для разных типов элементов (см. легенду)
- Ссылаться на легенду элементов нотации
- Добавить легенду на диаграмму для ясности
