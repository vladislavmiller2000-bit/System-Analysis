# 4. Требования к данным

## 4.1 Логическая модель данных

Логическая модель данных веб-приложения «Накарабине» представлена в нотации Crow's Foot и приведена к третьей нормальной форме (3НФ).

### Диаграмма ER-модели (Crow's Foot)

```
┌───────────────────┐                    ┌───────────────────────┐
│     Категория     │                    │         Товар         │
├───────────────────┤                    ├───────────────────────┤
│ * Номер категории │                    │ * Артикул товара      │
│   Название        │─────────||────<────│ o Номер категории (FK)│
│   Описание        │  содержит          │   Название            │
└───────────────────┘                    │   Описание            │
                                         │   Цена                │
     Одна категория                      │   Количество на складе│
     содержит много                      │   Фото                │
     товаров (1:M)                       └───────────────────────┘
                                                    │
                                                    │
                                         ─────||────<─────
                                                    │
                                         Один товар может
                                         быть в нескольких
                                         позициях (0:M)
                                                    │
                                                    ▼
┌───────────────────────────────────┐      ┌───────────────────────┐
│              Заказ                │      │    Позиция заказа     │
├───────────────────────────────────┤      ├───────────────────────┤
│ * Номер заказа                    │      │ * Номер заказа (FK)   │
│   Имя пользователя                │      │ * Артикул товара (FK) │
│   Телефон                         │──||──<──│   Количество          │
│   Электронный адрес               │содержит│   Цена за единицу     │
│   Почтовый индекс                 │      └───────────────────────┘
│   Страна                          │
│   Город                           │      Один заказ содержит
│   Улица                           │      от 1 до 50 позиций
│   Дом                             │      (согласно БП-3)
│   Квартира                        │
│   Способ оплаты                   │
│   Дата и время заказа             │
│   Дата и время доставки           │
│   Статус заказа                   │
│   Итоговая сумма                  │
│   Согласие на обработку данных    │
└───────────────────────────────────┘

Обозначения нотации Crow's Foot:
────||──── : Один (обязательный)
────O|──── : Ноль или один (необязательный)
────<───── : Много (со стороны "вилки")
  *        : Первичный ключ (PK)
  o        : Внешний ключ (FK)
```

### Описание связей (двусторонняя кардинальность)

В нотации Crow's Foot кардинальность указывается с обеих сторон связи в формате **(min, max)**:
- **min** — минимальное количество (0 = необязательно, 1 = обязательно)
- **max** — максимальное количество (1 = один, M = много)

---

#### 1. Категория ↔ Товар

**Как читается связь:** *Одна категория содержит много товаров; один товар принадлежит одной категории*

| Направление | Кардинальность | Описание |
|-------------|----------------|----------|
| **Категория → Товар** | (1, M) | Одна категория содержит **один или много** товаров |
| **Товар → Категория** | (1, 1) | Один товар принадлежит **ровно одной** категории |

**Тип связи:** Один-ко-многим (1:M)  
**Обязательность:** Связь обязательна с обеих сторон — товар не может существовать без категории, категория должна содержать хотя бы один товар.

---

#### 2. Товар ↔ Позиция заказа

**Как читается связь:** *Один товар может быть в нескольких позициях заказов; одна позиция содержит один товар*

| Направление | Кардинальность | Описание |
|-------------|----------------|----------|
| **Товар → Позиция заказа** | (0, M) | Один товар может быть в **нуле или многих** позициях заказов |
| **Позиция заказа → Товар** | (1, 1) | Одна позиция содержит **ровно один** товар |

**Тип связи:** Один-ко-многим (1:M)  
**Обязательность:** Товар может не быть заказан (0), но позиция обязательно содержит товар (1).

---

#### 3. Заказ ↔ Позиция заказа

**Как читается связь:** *Один заказ содержит от 1 до 50 позиций; одна позиция принадлежит одному заказу*

| Направление | Кардинальность | Описание |
|-------------|----------------|----------|
| **Заказ → Позиция заказа** | (1, 50) | Один заказ содержит **от 1 до 50** позиций (согласно БП-3) |
| **Позиция заказа → Заказ** | (1, 1) | Одна позиция принадлежит **ровно одному** заказу |

**Тип связи:** Один-ко-многим (1:M)  
**Обязательность:** Связь обязательна с обеих сторон — заказ не может быть пустым, позиция не существует без заказа.  
**Примечание:** Ограничение (1, 10) взято из бизнес-правила БП-3 спецификации.

| № | Сущность A | Сущность B | A → B | B → A | Как читается связь |
|---|------------|------------|-------|-------|-------------------|
| 1 | Категория | Товар | (1, M) | (1, 1) | Категория содержит товары |

### Обоснование нормализации (3НФ)

**Первая нормальная форма (1НФ):**
- Все атрибуты содержат атомарные (неделимые) значения
- Адрес разделён на компоненты: индекс, страна, город, улица, дом, квартира
- Нет повторяющихся групп — позиции заказа вынесены в отдельную сущность

**Вторая нормальная форма (2НФ):**
- Модель находится в 1НФ
- Все неключевые атрибуты полностью зависят от первичного ключа
- Выделена отдельная сущность «Позиция заказа» для связи многие-ко-многим между заказами и товарами

**Третья нормальная форма (3НФ):**
- Модель находится в 2НФ
- Отсутствуют транзитивные зависимости
- Категории вынесены в отдельную сущность (название категории не зависит от товара транзитивно)
- Цена за единицу в позиции заказа фиксирует цену на момент заказа (не транзитивная зависимость от товара)

---

### Обоснование выбора сущностей

**Почему 4 сущности, а не больше:**

1. **Пользователь** — не отдельная сущность, потому что:
   - В СТТ указано, что «личные аккаунты не предусмотрены» (ФТ-1.1.3)
   - Пользователь идентифицируется по номеру телефона при оформлении заказа
   - Данные пользователя (ФИО, телефон, email) — это атрибуты заказа

2. **Адрес доставки** — не отдельная сущность, потому что:
   - Нет бизнес-ключа (например, номера накладной)
   - Адрес привязан к конкретному заказу
   - Пользователь не может сохранять несколько адресов (нет личного кабинета)

3. **Способ оплаты** — не отдельная сущность, потому что:
   - Это справочное поле с фиксированными значениями
   - Два варианта: «Оплата банковской картой» и «Оплата наличными курьеру»
   - Хранится как атрибут заказа
