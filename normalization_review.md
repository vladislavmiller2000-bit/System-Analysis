# 🔍 УГЛУБЛЕННАЯ ПРОВЕРКА НОРМАЛИЗАЦИИ

## ❌ ПРОБЛЕМА 1: Таблица ТОВАР В ЗАКАЗЕ — НАРУШЕНИЕ 2НФ!

### Составной первичный ключ:
```
PK = (nomer_zakaza, artikul_tovara)
```

### Анализ атрибутов:
| Атрибут | Зависит от | Проблема |
|---------|-----------|---------|
| kolichestvo | (nomer_zakaza, artikul_tovara) | ✅ OK — зависит от ПОЛНОГО ключа |
| **tsena_za_edinitsu** | только от `artikul_tovara` | ❌ **ЧАСТИЧНАЯ ЗАВИСИМОСТЬ!** |
| **razmer_skidki** | только от `artikul_tovara` | ❌ **ЧАСТИЧНАЯ ЗАВИСИМОСТЬ!** |

### Почему это проблема?
- Цена товара зависит **только от артикула**, а не от заказа
- Размер скидки (вероятно) зависит **только от товара**, а не от заказа
- Если цена товара изменится, нужно обновить **все строки** этого товара во всех заказах
- Это ведет к **аномалиям обновления**!

### Решение — Извлечь отдельную таблицу:

```
ТОВАР (NEW)
├─ PK: artikul_tovara [VARCHAR]
├─ tsena_za_edinitsu [DECIMAL]
└─ razmer_skidki [DECIMAL]

ТОВАР_В_ЗАКАЗЕ
├─ PK, FK: nomer_zakaza [INT]
├─ PK, FK: artikul_tovara [VARCHAR] → ТОВАР
└─ kolichestvo [INT]  ← Только эта зависимость остается!
```

## ❌ ПРОБЛЕМА 2: Таблица ЗАКАЗ — Потенциальное нарушение 3НФ

### Анализ атрибутов дата/время:

```
PK = nomer_zakaza

Атрибуты:
- data_zakaza (дата)
- vremya_zakaza (время)
- status_zakaza (статус)
- ...
```

### Вопрос: Зависит ли `vremya_zakaza` от `data_zakaza`?

#### Сценарий 1: ДА, если это логически связанные атрибуты одного события
```
nomer_zakaza → data_zakaza → vremya_zakaza
                    ↑
            Транзитивная зависимость!
```
**Пример:**
- Заказ #1: Дата = 2025-01-15, Время = 14:30
- Если мы знаем дату, время все еще может быть 14:30, 15:45, 09:00 и т.д.
- Но логически, время БЕЗ даты бессмыслено!

#### Сценарий 2: НЕТ, если это независимые атрибуты
- Каждый товар заказывается независимо, и дата/время просто фиксируют момент
- Это нормально для 3НФ

### Проверка других неключевых атрибутов:
- `summa_zakaza` → это производная от товаров? ❓
- `summa_skidok` → это производная? ❓
- Если они производные (вычисляются на основе других), это может быть еще одна проблема!

## 🔧 ПОЛНАЯ ИСПРАВЛЕННАЯ МОДЕЛЬ (ПРАВИЛЬНАЯ 2НФ + 3НФ)

### Вариант A: Если дата/время логически независимы

```
ЗАКАЗ
├─ PK: nomer_zakaza [INT]
├─ data_zakaza [DATE]
├─ vremya_zakaza [TIME]
├─ status_zakaza [VARCHAR]
├─ sposob_oplaty [VARCHAR]
├─ summa_zakaza [DECIMAL]  ← если это НЕ производная
├─ summa_skidok [DECIMAL]  ← если это НЕ производная
└─ FK: id_polzovatelya [INT]

ТОВАР (NEW)
├─ PK: artikul_tovara [VARCHAR]
├─ tsena_za_edinitsu [DECIMAL]
└─ razmer_skidki [DECIMAL]

ТОВАР_В_ЗАКАЗЕ
├─ PK, FK: nomer_zakaza [INT]
├─ PK, FK: artikul_tovara [VARCHAR]
└─ kolichestvo [INT]

СВЕДЕНИЯ_О_ДОСТАВКЕ
├─ PK, FK: nomer_zakaza [INT]
├─ sposob_dostavki [VARCHAR]
├─ data_dostavki [DATE]
└─ FK: id_adresa [INT]

АДРЕС
├─ PK: id_adresa [INT]
├─ strana [VARCHAR]
├─ gorod [VARCHAR]
├─ ulitsa [VARCHAR]
├─ nomer_doma [VARCHAR]
├─ kvartira [VARCHAR]
└─ pochtovyy_indeks [VARCHAR]
```

### Вариант B: Если дата/время создают транзитивную зависимость

```
ЗАКАЗ
├─ PK: nomer_zakaza [INT]
├─ FK: id_vremeni_zakaza [INT]  → VREMYA_ZAKAZA (объединить дату и время)
├─ status_zakaza [VARCHAR]
├─ sposob_oplaty [VARCHAR]
├─ summa_zakaza [DECIMAL]
├─ summa_skidok [DECIMAL]
└─ FK: id_polzovatelya [INT]

VREMYA_ZAKAZA (NEW)
├─ PK: id_vremeni_zakaza [INT]
├─ data_zakaza [DATE]
└─ vremya_zakaza [TIME]  ← или единый TIMESTAMP
```
