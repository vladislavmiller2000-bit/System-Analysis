# Нормализация ER-модели до 3НФ

## ЧТО ТАКОЕ НОРМАЛИЗАЦИЯ?

**Нормализация** — это процесс **структурирования данных** в таблицах для:
- ✅ Избежать повторения данных (дублирование)
- ✅ Избежать неконсистентности (противоречия в данных)
- ✅ Улучшить производительность запросов
- ✅ Облегчить обновление и удаление данных

Есть **три основных уровня** нормализации (нормальные формы):
1. **1НФ** (Первая нормальная форма)
2. **2НФ** (Вторая нормальная форма)
3. **3НФ** (Третья нормальная форма)

---

## 1. ПЕРВАЯ НОРМАЛЬНАЯ ФОРМА (1НФ)

### Определение

**Таблица находится в 1НФ, если:**
- ✅ Все значения в колонках **атомарны** (неделимы)
- ✅ **Нет повторяющихся групп** атрибутов
- ✅ Каждое поле содержит **одно значение**
- ✅ Нет списков, массивов, вложенных таблиц в одной ячейке

### Нарушения 1НФ (примеры):

```
❌ НЕПРАВИЛЬНО (нарушает 1НФ):
Таблица: ЗАКАЗ_НЕПРАВИЛЬНО
┌─────┬──────────────────────────────────────┐
│ ID  │ Товары (список в одной ячейке)      │
├─────┼──────────────────────────────────────┤
│ 1   │ RUB001 (2 шт), RUB002 (1 шт)         │
│ 2   │ RUB003 (3 шт), RUB001 (1 шт)         │
└─────┴──────────────────────────────────────┘

Проблемы:
- Товары хранятся как список в одной ячейке
- Нельзя обновить цену одного товара в заказе 2
- Сложно поиск и фильтрацию по товарам
- Нарушение атомарности
```

### Как привести к 1НФ:

**Решение**: Создать отдельную таблицу для повторяющихся групп

```
✅ ПРАВИЛЬНО (в 1НФ):
Таблица: ЗАКАЗ
┌─────┬──────────┐
│ ID  │ Дата     │
├─────┼──────────┤
│ 1   │ 20.12    │
│ 2   │ 21.12    │
└─────┴──────────┘

Таблица: ТОВАР_В_ЗАКАЗЕ
┌────────┬──────────┬─────────┐
│ ID_зак │ Артикул  │ Кол-во  │
├────────┼──────────┼─────────┤
│ 1      │ RUB001   │ 2       │
│ 1      │ RUB002   │ 1       │
│ 2      │ RUB003   │ 3       │
│ 2      │ RUB001   │ 1       │
└────────┴──────────┴─────────┘

Преимущества:
- Каждое значение атомарно
- Можно легко добавлять/удалять товары
- Возможна фильтрация по товарам
- Консистентность данных
```

## 2. ВТОРАЯ НОРМАЛЬНАЯ ФОРМА (2НФ)

### Определение

**Таблица находится в 2НФ, если:**
- ✅ Находится в **1НФ**
- ✅ Все **неключевые атрибуты зависят от ВСЕГО первичного ключа** (не от его части)
- ✅ **Нет частичных зависимостей**

### Что такое частичная зависимость?

```
ПРИМЕР ЧАСТИЧНОЙ ЗАВИСИМОСТИ (нарушение 2НФ):

Таблица: ТОВАР_В_ЗАКАЗЕ (неправильно)
┌────────┬──────────┬──────────┬──────────────┐
│ ID_зак │ Артикул  │ Кол-во   │ Название_тов │  ← Зависит только от Артикула
├────────┼──────────┼──────────┼──────────────┤    (не от полного ключа)
│ 1      │ RUB001   │ 2        │ Рубашка      │
│ 1      │ RUB002   │ 1        │ Штаны        │
│ 2      │ RUB001   │ 1        │ Рубашка      │ ← Одинаковое название
└────────┴──────────┴──────────┴──────────────┘

Проблемы:
- Название товара повторяется (дублирование)
- Если изменить название RUB001, нужно обновить ВСЕ записи
- Неконсистентность: если обновим неправильно
- Избыточное хранение данных

Первичный ключ: (ID_зак, Артикул)
Зависимость:
- Кол-во зависит от ПОЛНОГО ключа ✅
- Название_тов зависит ТОЛЬКО от Артикула (частичная ❌)
```

### Как привести к 2НФ:

**Решение**: Выделить атрибуты в отдельную таблицу

```
✅ ПРАВИЛЬНО (в 2НФ):
Таблица: ТОВАР_В_ЗАКАЗЕ
┌────────┬──────────┬─────────┐
│ ID_зак │ Артикул  │ Кол-во  │
├────────┼──────────┼─────────┤
│ 1      │ RUB001   │ 2       │
│ 1      │ RUB002   │ 1       │
│ 2      │ RUB003   │ 3       │
│ 2      │ RUB001   │ 1       │
└────────┴──────────┴─────────┘

Таблица: ТОВАР
┌──────────┬──────────────┐
│ Артикул  │ Название_тов │
├──────────┼──────────────┤
│ RUB001   │ Рубашка      │
│ RUB002   │ Штаны        │
│ RUB003   │ Футболка     │
└──────────┴──────────────┘

Преимущества:
- Нет дублирования (название только один раз)
- Легко обновить название (в одном месте)
- Консистентность (нет противоречий)
- Экономия памяти
```

## 3. ТРЕТЬЯ НОРМАЛЬНАЯ ФОРМА (3НФ)

### Определение

**Таблица находится в 3НФ, если:**
- ✅ Находится в **2НФ**
- ✅ **Нет транзитивных зависимостей**
- ✅ Все неключевые атрибуты зависят **только от первичного ключа**
- ✅ Неключевые атрибуты **НЕ зависят друг от друга**

### Что такое транзитивная зависимость?

```
ПРИМЕР ТРАНЗИТИВНОЙ ЗАВИСИМОСТИ (нарушение 3НФ):

Таблица: ЗАКАЗ (неправильно)
┌────────┬──────────┬──────────┬──────────┐
│ ID     │ Дата     │ Город    │ Почт_код │  ← Зависит от Города
├────────┼──────────┼──────────┼──────────┤    (не от ID заказа)
│ 1      │ 20.12    │ Москва   │ 119991   │
│ 2      │ 21.12    │ Москва   │ 119991   │  ← Одинаковые город и код
│ 3      │ 22.12    │ СПБ      │ 190000   │
└────────┴──────────┴──────────┴──────────┘

Проблемы:
- Почт_код зависит от Города, а не от ID заказа
- Дублирование (Москва + 119991 повторяется)
- Если изменится почтовый код Москвы, нужно обновить ВСЕ заказы
- Неконсистентность: можно создать противоречие (Москва = 999999)

Цепочка зависимостей:
ID заказа → Город → Почт_код (транзитивная зависимость ❌)
```

### Как привести к 3НФ:

**Решение**: Выделить зависимые атрибуты в отдельную таблицу

```
✅ ПРАВИЛЬНО (в 3НФ):
Таблица: ЗАКАЗ
┌────────┬──────────┬──────────┐
│ ID     │ Дата     │ ID_города│
├────────┼──────────┼──────────┤
│ 1      │ 20.12    │ 101      │
│ 2      │ 21.12    │ 101      │
│ 3      │ 22.12    │ 102      │
└────────┴──────────┴──────────┘

Таблица: ГОРОД
┌──────────┬──────────┬──────────┐
│ ID_города│ Название │ Почт_код │
├──────────┼──────────┼──────────┤
│ 101      │ Москва   │ 119991   │
│ 102      │ СПБ      │ 190000   │
└──────────┴──────────┴──────────┘

Преимущества:
- Почтовый код хранится один раз
- Легко обновить (в одном месте)
- Нет транзитивных зависимостей
- Консистентность (Москва всегда = 119991)
- Экономия памяти
```

## ВИЗУАЛЬНОЕ ПРЕДСТАВЛЕНИЕ

### Зависимости (диаграмма):

```
1НФ (базовый уровень):
┌──────────────────┐
│ Атомарные данные │
└──────────────────┘

2НФ (избегаем частичных зависимостей):
┌──────────────────┐
│ Полные зависимости│  Все атрибуты зависят от
│  от ключа        │  ВСЕГО первичного ключа
└──────────────────┘

3НФ (избегаем транзитивных зависимостей):
┌──────────────────┐
│ Только от ключа  │  Атрибуты зависят ТОЛЬКО
│ (не от друг от   │  от первичного ключа,
│  друга)          │  не друг от друга
└──────────────────┘
```

### ✅ ПРОВЕРКА 3НФ

**Вопрос**: Есть ли транзитивные зависимости (атрибуты зависят друг от друга, а не от PK)?

#### Таблица ЗАКАЗ

```
Проверка транзитивных зависимостей:
- data_zakaza зависит ТОЛЬКО от nomer_zakaza ✅
- vremya_zakaza зависит ТОЛЬКО от nomer_zakaza ✅
- status_zakaza зависит ТОЛЬКО от nomer_zakaza ✅
- sposob_oplaty зависит ТОЛЬКО от nomer_zakaza ✅
- id_polzovatelya зависит ТОЛЬКО от nomer_zakaza ✅

Есть ли зависимости между неключевыми атрибутами?
- data_zakaza → fremya_zakaza? НЕТ
- status_zakaza → sposob_oplaty? НЕТ
- Нет цепочек типа: A → B → C
```
**Статус**: ✅ 3НФ (нет транзитивных зависимостей)

#### Таблица ТОВАР_В_ЗАКАЗЕ

```
Проверка транзитивных зависимостей:
- kolichestvo → зависит от (nomer_zakaza, artikul_tovara) ✅
- tsena_za_edinitsu → зависит от (nomer_zakaza, artikul_tovara) ✅
```

## КАК ПРИВЕСТИ К 3НФ

### Решение: Выделить АДРЕС в отдельную таблицу

```
ВАРИАНТ 1 (рекомендуемый): Создать таблицу АДРЕС

Таблица: АДРЕС
┌──────────┬──────────┬──────────┬──────────┬──────────┬─────────────┐
│ id_adresa│ strana   │ gorod    │ ulitsa   │ nomer_d  │ pochtovyy_i │
│ (PK)     │          │          │          │ oma      │ ndeks       │
├──────────┼──────────┼──────────┼──────────┼──────────┼─────────────┤
│ 1        │ Россия   │ Москва   │ Ленинск.│ 45       │ 119991      │
│ 2        │ Россия   │ Москва   │ Красная │ 1        │ 109012      │
│ 3        │ Россия   │ СПБ      │ Невский │ 28       │ 190000      │
└──────────┴──────────┴──────────┴──────────┴──────────┴─────────────┘

Таблица: SVEDENIYA_O_DOSTAVKE (исправленная)
┌────────┬──────────────┬──────────────┬──────────┐
│ nomer_ │ sposob_      │ data_        │ id_adresa│
│ zakaza │ dostavki     │ dostavki     │ (FK)     │
│ (PK,FK)│              │              │          │
├────────┼──────────────┼──────────────┼──────────┤
│ 1      │ курьер       │ 2024-12-23   │ 1        │
│ 2      │ почта        │ 2024-12-25   │ 2        │
│ 3      │ самовывоз    │ NULL         │ 3        │
└────────┴──────────────┴──────────────┴──────────┘

Новая связь:
SVEDENIYA_O_DOSTAVKE (N) ─── (1) ADRES
```

**Преимущества**:
- ✅ Адрес хранится один раз
- ✅ Нет транзитивных зависимостей
- ✅ Легко обновить адрес
- ✅ Можно переиспользовать адреса для разных целей
- ✅ Полная **3НФ** ✅
