# ER-диаграмма в нотации Crow's Foot

## 1. ЧТО ТАКОЕ CROW'S FOOT?

**Crow's Foot (Вороньи лапки)** — это физическая ER-модель, которая:
- Близка к реальной реализации баз данных
- Показывает **таблицы** вместо абстрактных сущностей
- Показывает **атрибуты как колонки** таблицы
- Использует специальные **символы на линиях связи** для кардинальности

### Отличия от нотации Чена:

| Аспект | Чена | Crow's Foot |
|--------|------|------------|
| **Сущность** | Абстрактный прямоугольник | Таблица БД |
| **Атрибуты** | Овалы вокруг сущности | Колонки внутри таблицы |
| **Первичный ключ** | Подчеркнутый овал | PK перед названием |
| **Внешний ключ** | Овал помеченный (FK) | FK перед названием |
| **Связь** | Ромбик + линия с цифрами | Линия с вороньими лапками |
| **Уровень** | Логический (концептуальный) | Физический (реализационный) |

## 2. СТРУКТУРА ТАБЛИЦЫ В CROW'S FOOT

```
┌──────────────────────────────────┐
│           ЗАКАЗ                  │  ← Название таблицы
├──────────────────────────────────┤
│ PK  № заказа            [INT]    │  ← Первичный ключ (Primary Key)
│     Дата заказа         [DATE]   │  ← Обычный атрибут
│     Время заказа        [TIME]   │
│     Статус заказа       [VARCHAR]│
│     Способ оплаты       [VARCHAR]│
│     Сумма заказа        [DECIMAL]│  ← Вычисляемое поле (производное)
│     Сумма скидок        [DECIMAL]│
│ FK  ID Пользователя     [INT]    │  ← Внешний ключ (Foreign Key)
└──────────────────────────────────┘
```

### Элементы таблицы:

1. **Название таблицы** (заголовок)
   - Обычно в **единственном числе** (ЗАКАЗ, не ЗАКАЗЫ)
   - **PascalCase** или UPPERCASE
   - Должно быть существительным

2. **Первичный ключ (PK)**
   - Отмечается буквами **PK** перед названием
   - **Уникально идентифицирует** каждую запись
   - В таблице может быть только **один простой PK**
   - Может быть **составным** (несколько полей)

3. **Внешний ключ (FK)**
   - Отмечается буквами **FK** перед названием
   - **Ссылается на PK другой таблицы**
   - Создаёт связь между таблицами

4. **Обычные атрибуты**
   - Не отмечаются PK или FK
   - Содержат данные

5. **Тип данных** (в квадратных скобках)
   - INT, VARCHAR, DATE, TIME, DECIMAL и т.д.

## 3. КАРДИНАЛЬНОСТЬ В CROW'S FOOT

Кардинальность показывается на **концах линий связи** между таблицами.

### Символы кардинальности:

#### 3.1 Один к одному (1:1)

```
ТАБЛИЦА_A          ТАБЛИЦА_B
┌────────┐         ┌────────┐
│ id (PK)├────────┤ id (PK)│
│        │  1   1 │        │
└────────┘         └────────┘
```

- **1** на обеих концах
- Один экземпляр A связан с **ровно одним** экземпляром B
- **Пример**: ЗАКАЗ (1) ─── (1) СВЕДЕНИЯ О ДОСТАВКЕ
  - Один заказ имеет ровно одно сведение о доставке
  - Одно сведение о доставке принадлежит ровно одному заказу

#### 3.2 Один ко многим (1:N)

```
ТАБЛИЦА_A               ТАБЛИЦА_B
┌────────┐              ┌──────────────┐
│ id (PK)├─────────────┤║ id (PK, FK)  │
│        │   1    N    │  ...          │
└────────┘              └──────────────┘
```

- **1** на одном конце, **N** на другом
- Вороньи лапки (├─) со стороны "N"
- Один экземпляр A связан с **многими** экземплярами B
- B может существовать только если связан с A (идентифицирующая связь)
- **Пример**: ЗАКАЗ (1) ─── (N) ТОВАР В ЗАКАЗЕ
  - Один заказ содержит **много** товаров
  - Каждый товар в заказе принадлежит **ровно одному** заказу

#### 3.3 Много ко многим (M:N)

```
ТАБЛИЦА_A          АССОЦИАТИВНАЯ_ТАБЛИЦА          ТАБЛИЦА_B
┌────────┐        ┌──────────────────┐            ┌────────┐
│ id (PK)├───────┤ id_A (PK, FK)    │───────────┤ id (PK)│
│        │  1  N │ id_B (PK, FK)    │ N    1    │        │
└────────┘        └──────────────────┘            └────────┘
```

- Обе стороны имеют **вороньи лапки (N)**
- Требует **ассоциативной таблицы** для разрешения M:N
- Ассоциативная таблица содержит **составной первичный ключ** из двух FK
- **Пример**: ЗАКАЗ (M) ─── (N) ТОВАР через таблицу ТОВАР В ЗАКАЗЕ

### Расширенные символы кардинальности:

```
─────  1      (Ровно один)

├─ или ├─<    N (Много, вороньи лапки)

o─────  0..1  (Ноль или один, опциональное)

o├─     0..N  (Ноль или много, опциональное)
```

## 4. ТИПЫ СВЯЗЕЙ В CROW'S FOOT

### 4.1 Идентифицирующая связь (Identifying Relationship)

```
ЗАКАЗ                    ТОВАР В ЗАКАЗЕ
┌──────────┐            ┌─────────────────┐
│ PK id    ├──────────┤║ PK, FK id_zakas │ ← Комбинированный ключ
│          │    1  N  │  PK artikul      │
└──────────┘            └─────────────────┘
```

**Свойства**:
- Обычно **сплошная линия**
- Дочерняя таблица **зависит от родительской**
- PK дочерней таблицы **включает FK родительской**
- Дочерняя запись **не может существовать без родительской**

**Пример в нашей модели**:
- ТОВАР В ЗАКАЗЕ не может существовать без ЗАКАЗА
- Полный ключ ТОВАРА В ЗАКАЗЕ = (№ заказа, Артикул товара)

### 4.2 Неидентифицирующая связь (Non-identifying Relationship)

```
ПОЛЬЗОВАТЕЛЬ            ЗАКАЗ
┌──────────┐            ┌──────────┐
│ PK id    ├───────────┤ PK id    │
│          │    1   N  │ FK id_user│ ← FK не входит в PK
└──────────┘            └──────────┘
```

**Свойства**:
- Обычно **пунктирная линия**
- Дочерняя таблица **независима** (имеет свой PK)
- FK дочерней таблицы **НЕ входит** в её PK
- Дочерняя запись **может существовать без родительской** (опционально)

## 5. НАША МОДЕЛЬ: ТАБЛИЦЫ И СВЯЗИ

### Таблица 1: ЗАКАЗ (независимая сущность)

```
┌─────────────────────────────────────────────────────┐
│ ЗАКАЗ                                               │
├─────────────────────────────────────────────────────┤
│ PK  № заказа          [INT]       ← Уникальный      │
│     Дата заказа       [DATE]                        │
│     Время заказа      [TIME]                        │
│     Статус заказа     [VARCHAR]   ← Допустимые      │
│                                    значения:         │
│                                    'оформлен'        │
│                                    'оплачен'         │
│                                    'доставлен'       │
│                                    'отменён'         │
│     Способ оплаты     [VARCHAR]   ← Способ оплаты   │
│     Сумма заказа      [DECIMAL]   ← Производное     │
│                                    (вычисляется)    │
│     Сумма скидок      [DECIMAL]   ← Производное     │
│ FK  ID Пользователя   [INT]       ← Ссылка на       │
│                                    ПОЛЬЗОВАТЕЛЯ     │
└─────────────────────────────────────────────────────┘
```

**Валидация**:
- ✅ Название: ЗАКАЗ (единственное число, PascalCase)
- ✅ PK: № заказа (уникальный идентификатор)
- ✅ FK: ID Пользователя (связь с пользователем)
- ✅ Типы данных подобраны логично

---

### Таблица 2: ТОВАР В ЗАКАЗЕ (зависимая сущность)

```
┌──────────────────────────────────────────────────────┐
│ ТОВАР В ЗАКАЗЕ                                       │
├──────────────────────────────────────────────────────┤
│ PK, FK  № заказа      [INT]       ← Часть PK и FK   │
│ PK      Артикул       [VARCHAR]   ← Часть PK        │
│         Количество    [INT]       ← Кол-во единиц   │
│         Цена за       [DECIMAL]   ← Цена товара     │
│         единицу                     на момент заказа│
│         Размер        [DECIMAL]   ← Процент скидки  │
│         скидки                      (из акции)       │
└──────────────────────────────────────────────────────┘
```

**Валидация**:
- ✅ Название: ТОВАР В ЗАКАЗЕ (описательное)
- ✅ Составной PK: (№ заказа, Артикул)
  - Один заказ + один товар = уникальная позиция
- ✅ FK: № заказа ссылается на ЗАКАЗ
- ✅ Зависимая от ЗАКАЗА (идентифицирующая связь)

**Почему составной PK?**
- В одном заказе может быть несколько товаров
- Каждый товар в заказе уникален по комбинации (заказ + артикул)
- Нельзя использовать только артикул, так как один товар может быть в разных заказах

---

### Таблица 3: СВЕДЕНИЯ О ДОСТАВКЕ (зависимая сущность)

```
┌────────────────────────────────────────────────────┐
│ СВЕДЕНИЯ О ДОСТАВКЕ                                │
├────────────────────────────────────────────────────┤
│ PK, FK  № заказа      [INT]       ← Полный PK      │
│         Способ        [VARCHAR]   ← Способ:        │
│         доставки                    'курьер'        │
│                                     'самовывоз'     │
│                                     'почта'         │
│         Дата доставки [DATE]       ← Ожидаемая дата│
│         Страна        [VARCHAR]    ← Адрес доставки│
│         Город         [VARCHAR]    │ (составной)    │
│         Улица         [VARCHAR]    │                │
│         Номер дома    [VARCHAR]    │                │
│         Квартира      [VARCHAR]    │                │
│         Почтовый      [VARCHAR]    │                │
│         индекс                      │                │
└────────────────────────────────────────────────────┘
```

**Валидация**:
- ✅ Название: СВЕДЕНИЯ О ДОСТАВКЕ (описывает назначение)
- ✅ PK = FK: № заказа
  - Один заказ = одно сведение о доставке (1:1)
  - PK одновременно является FK (очень узкая связь)
- ✅ Адрес — составной атрибут (разбит на компоненты)
- ✅ Зависимая от ЗАКАЗА (идентифицирующая связь)

**Почему PK = FK?**
- Каждый заказ имеет **ровно одно** сведение о доставке (1:1)
- Сведение о доставке не может существовать без заказа
- Поэтому PK СВЕДЕНИЙ О ДОСТАВКЕ = FK на ЗАКАЗ

## 6. СВЯЗИ МЕЖДУ ТАБЛИЦАМИ

### Связь 1: ЗАКАЗ ─────── ТОВАР В ЗАКАЗЕ

```
      ЗАКАЗ                     ТОВАР В ЗАКАЗЕ
    ┌────────┐                ┌──────────────┐
    │ id (PK)├────────────────┤║id (PK, FK)  │
    │        │      1    N    │ artikul (PK) │
    └────────┘                └──────────────┘
```

- **Тип**: Идентифицирующая (сплошная линия)
- **Кардинальность**: 1:N
- **Чтение**: "Один ЗАКАЗ содержит много ТОВАРОВ В ЗАКАЗЕ"
- **Свойства**:
  - Обязательная связь
  - ТОВАР В ЗАКАЗЕ не может существовать без ЗАКАЗА
  - При удалении ЗАКАЗА удаляются все его ТОВАРЫ

### Связь 2: ЗАКАЗ ─────── СВЕДЕНИЯ О ДОСТАВКЕ

```
      ЗАКАЗ                    СВЕДЕНИЯ О ДОСТАВКЕ
    ┌────────┐                ┌──────────────┐
    │ id (PK)├────────────────┤║id (PK, FK)  │
    │        │      1    1    │ ...          │
    └────────┘                └──────────────┘
```

- **Тип**: Идентифицирующая (сплошная линия)
- **Кардинальность**: 1:1
- **Чтение**: "Один ЗАКАЗ имеет одно СВЕДЕНИЕ О ДОСТАВКЕ"
- **Свойства**:
  - Обязательная связь
  - Очень тесная связь (PK = FK)
  - Можно было бы объединить в одну таблицу (но разделено для наглядности)

## 7. ПРАВИЛА ВАЛИДАЦИИ CROW'S FOOT

### 7.1 Правила для названий

**Таблицы (Entity names)**:
- ✅ Единственное число: ЗАКАЗ (не ЗАКАЗЫ)
- ✅ Существительное: ТОВАР В ЗАКАЗЕ
- ✅ PascalCase или UPPERCASE: ЗАКАЗ, ТОВАР_В_ЗАКАЗЕ
- ✅ На русском или английском (не смешивать)
- ❌ Глаголы, прилагательные в названии таблицы

**Колонки (Attribute names)**:
- ✅ Существительное: № заказа, Дата доставки
- ✅ Единственное число или значимые существительные
- ✅ snake_case или camelCase: nomer_zakaza, nomerZakaza
- ✅ Должны быть информативными
- ❌ Сокращения без смысла: "z" вместо "zakas"

**Примеры на русском**:
```
✅ Хорошо:
- ЗАКАЗ
- ТОВАР_В_ЗАКАЗЕ
- Дата_доставки
- Способ_оплаты

❌ Плохо:
- ЗАКАЗЫ (множественное число)
- ДОСТАВИТЬ (глагол)
- Доставки (множественное число)
- т (сокращение без смысла)
```

### 7.2 Правила для ключей

**Первичный ключ (PK)**:
- ✅ Обязателен в каждой таблице
- ✅ Должен быть уникальным
- ✅ Не может быть NULL
- ✅ Может быть простым (одно поле) или составным (несколько полей)
- ✅ Обычно: id, nomer, kod (не просто "id", а описательный)

**Внешний ключ (FK)**:
- ✅ Ссылается на PK другой таблицы
- ✅ Должен иметь то же имя или похожее (например, id_zakaza ← ЗАКАЗ.id)
- ✅ Может быть NULL (если связь опциональна)
- ✅ Обозначается буквами FK перед названием

### 7.3 Правила для связей

- ✅ Кардинальность указывается на обеих концах линии
- ✅ Линия соединяет FK одной таблицы с PK другой
- ✅ Идентифицирующие связи (зависимые сущности) — FK входит в PK
- ✅ Неидентифицирующие связи (независимые сущности) — FK не входит в PK
- ❌ Не должно быть цикличных зависимостей (A → B → A)

### 7.4 Правила для типов данных

- ✅ INT для целых чисел: ID, количество, возраст
- ✅ VARCHAR/TEXT для текста: имя, адрес, описание
- ✅ DATE для дат: дата рождения, дата доставки
- ✅ TIME для времени: время заказа
- ✅ DECIMAL/NUMERIC для денег: цена, сумма (не FLOAT!)
- ✅ BOOLEAN для да/нет значений
- ❌ Не использовать FLOAT для финансовых данных (проблемы с округлением)

## 8. ИНСТРУКЦИИ ДЛЯ DIAGRAMS.NET

### Как создать диаграмму Crow's Foot:

#### Шаг 1: Создание таблицы
1. Вставить **прямоугольник** (Rectangle)
2. Разделить на две части: заголовок и тело
   - Заголовок: название таблицы
   - Тело: список колонок
3. Форматирование:
   - Заголовок: синий фон, белый текст, жирный
   - Тело: светло-серый фон, чёрный текст

#### Шаг 2: Добавление колонок
1. Использовать **текст** для каждой колонки
2. Формат строк:
   ```
   PK  № заказа               [INT]
   FK  ID Пользователя        [INT]
       Дата заказа            [DATE]
   ```
3. Цветовое выделение:
   - PK: жёлтый фон
   - FK: зелёный фон
   - Обычные: белый фон

#### Шаг 3: Создание связей
1. Вставить **линию** (Line) между таблицами
2. Добавить **символы кардинальности** (текст или специальные символы)
3. Нарисовать **вороньи лапки** (три короткие линии в углов) для N
4. Для 1:1 — просто линия на обоих концах

#### Шаг 4: Добавление легенды
1. Создать таблицу или текстовое поле с описанием символов
2. Включить все используемые обозначения

## 10. СРАВНЕНИЕ НАШЕЙ МОДЕЛИ

### До нормализации (логическая модель):

```
ЗАКАЗ (множественный атрибут)
├── Товары в заказе (Артикул, Количество, Цена, Скидка)
├── Сведения о доставке (Способ, Дата, Адрес)
└── Производные (Сумма, Скидки)
```

### После преобразования в Crow's Foot (физическая модель):

```
ЗАКАЗ (таблица)
├── ТОВАР_В_ЗАКАЗЕ (зависимая таблица, составной PK)
└── СВЕДЕНИЯ_О_ДОСТАВКЕ (зависимая таблица, PK=FK)
```

**Выводы**:
- ✅ Множественный атрибут стал отдельной таблицей
- ✅ Составной атрибут разбит на колонки в таблице
- ✅ Производные атрибуты остаются как вычисляемые поля
- ✅ Связи чётко обозначены PK-FK
